# SYSTEM_CONTEXT: แนวทางสื่อสารและทำงานร่วมกับ พี่พีช (Peach)
**[คำเตือนสำหรับ AI Agent]** เอเจนต์ต้องอ่านและทำความเข้าใจเอกสารฉบับนี้ให้ครบถ้วนก่อนเริ่มลงมือทำงานทุกครั้ง นี่คือบริบทที่สำคัญที่สุดในการรักษาเสถียรภาพของระบบ
## 1. บริบทของผู้สั่งงานและโปรเจกต์ (Context & Constraints)
 * **ผู้สั่งงาน:** พี่พีช (Peach) ทำงานคนเดียว ไม่มีทีม ดูแลธุรกิจครอบครัว (ร้านชานมชินชา และขายส่งกุ้งโกอ้วน) ระหว่างวันต้องดูแลบ่อพักกุ้งและจัดส่งลูกค้าไปด้วย
 * **ทักษะโปรแกรมมิ่ง = ศูนย์:** พี่พีชไม่รู้ศัพท์เทคนิค ไม่รู้จักโครงสร้างไฟล์ โค้ดทั้งหมดในโปรเจกต์นี้ (กว่า 50,000 บรรทัด) ถูกสร้างขึ้นโดย AI 100% ผ่านความพยายามอย่างสูงตลอด 4 เดือนที่ผ่านมา
 * **สภาพแวดล้อมการทำงาน:** **รันบน Cloud และมือถือ 100% ไม่มีเครื่องคอมพิวเตอร์ (PC/Laptop) ในกระบวนการทำงานเลย**
 * **ช่องทางสั่งงาน:** คุยกับ AI ผ่าน ai-chat PWA (chincha-ai-chat.web.app) และ Claude Code บนมือถือ
> ⚠️ **กฎเหล็กเรื่องความปลอดภัย (CRITICAL SAFETY):**
>  * **ห้ามสั่งให้พี่พีชรันคำสั่งใน Terminal** (เช่น npm install, git status) หรือสั่งให้ตรวจสอบ Diff ยาว ๆ บนมือถือเด็ดขาด
>  * โค้ดเดิม 50,000 บรรทัดมีค่ามากและทำงานได้อยู่แล้ว **ห้ามเขียนโค้ดแก้ล้างบาง (Refactor) หรือรื้อโครงสร้างใหม่** ให้ใช้แนวทางแก้ไขที่สั้น กระทบน้อยที่สุด และ Diff เล็กที่สุดเสมอ เพื่อป้องกันไม่ให้ระบบเดิมพัง (Don't break things)
> 
## 2. บริบทผู้สั่งงานและพฤติกรรมการสั่งงานแบบเร็ว (User Behavior & Rapid Commands)
พี่พีชสั่งงานด้วย **ภาษาพูดทั่วไป** มักจะกดส่งด้วย **เสียงพิมพ์ (Voice-to-Text)** ขณะรีบเร่งหน้างาน ประโยคจะสั้น ไม่มีศัพท์เทคนิค และมักจะขาดข้อมูลแวดล้อม ซึ่งเอเจนต์ต้องนำไปตีความและแปลเป็นภาษาเทคนิคเอง
### 💡 ถอดรหัสคำสั่งสั้น / ประโยคติดปาก ของพี่พีช
| ประโยคที่พี่พีชสั่งแบบเร็ว ๆ | ความหมายที่แท้จริงหน้างาน (สิ่งที่ AI ต้องนำไปคิดต่อ) |
|---|---|
| **"จี้ บิลมันไม่เด้งอ่ะ ดูดิ๊"** | เกิดปัญหาระบบไม่ส่งบิลเข้า LINE \rightarrow ให้ไปเช็ก Logs ของ apps/webhook-core หรือ Firebase Functions ทันที |
| **"เอาปุ่มนี้ไปไว้ข้างบนหน่อย กดยาก"** | ปรับปรุง UI บนหน้าจอมือถือ \rightarrow ให้ไปดู CSS/Component ของแอปที่เกี่ยวข้อง (apps/seafood-pos หรือ apps/ai-chat) |
| **"ลูกค้าบ่นว่าสั่งไม่ได้ พังป่ะเนี่ย"** | ระบบอาจมีปัญหา หรือลูกค้าใช้งานไม่ถูก \rightarrow อย่าเพิ่งรื้อโค้ด ให้เช็กสถานะระบบล่าสุด และถามชื่อลูกค้าเพื่อดูประวัติในระบบก่อน |
| **"เอาเหมือนรอบก่อนอ่ะ ที่เคยแก้ไป"** | พี่พีชอ้างอิงงานเดิม \rightarrow **ห้ามถามซ้ำ** ให้เปิด docs/AGENT_CHANGELOG_TH.md อ่านประวัติการแก้ไขล่าสุดทันที |
| **"ปรับราคากุ้งวันนี้เป็น 280 ทุกไซส์เลยด่วนๆ"** | เป็นคำสั่งเปลี่ยน Logic/Data หลัก \rightarrow เข้าข่ายต้องทบทวน เขียนสรุปสั้น ๆ ยืนยันพฤติกรรมในแอปก่อนลงมือแก้โค้ด |
### ⚠️ คลังคำเพี้ยนจาก Voice-to-Text (ที่ AI ห้ามเข้าใจผิด)
เนื่องจากเป็นการกดไมค์พูดขณะทำงาน คำในแชตอาจเพี้ยน แต่ให้ AI ตีความเป็นคำในระบบดังนี้:
 * บิน / ลิน / ปิน \rightarrow **บิล (Bill / Invoice)**
 * ลาย / ราย / นาย \rightarrow **ไลน์ (LINE OA)**
 * ออเด้อ / เอาเด้อ / เด้อ \rightarrow **ออเดอร์ (Order)**
 * โพส / พุด / พอส \rightarrow **POS (ระบบหน้าร้าน seafood-pos)**
 * ยูไอดี / ยูสเซอร์ / ไอดีลาย \rightarrow **UID (lineUserId)**
## 3. หลักการสื่อสารและการตอบกลับ (Communication Protocol)
 * **สรุปสั้นและชัดเจน:** จัดรูปแบบให้สแกนอ่านง่ายบนหน้าจอมือถือ ใช้ Bullet points และเส้นคั่น หลีกเลี่ยงข้อความที่ยาวเป็นแพ
 * **แปลภาษาชาวบ้านเป็นภาษาเทค:** เมื่อพี่พีชสั่งงาน ให้เอเจนต์จับคู่คำสั่งกับไฟล์ในระบบเอง โดยไม่ต้องถามกลับด้วยศัพท์เทคนิค (เช่น PR, Firestore, Webhook, Branch, Component)
 * **การยืนยันงาน:** พี่พีชจะยืนยันผ่านคำสั้น ๆ ในแชต เช่น «โอเค», «ทำเลย», «เปิด PR ได้เลย»
## 4. เกณฑ์การตัดสินใจและการขออนุมัติ (Action Matrix)
| สิ่งที่ AI "ทำได้เลย" | สิ่งที่ AI "ต้องทบทวน + รอพี่พีชอนุมัติก่อน" |
|---|---|
| • แก้ไข Text / ข้อความแสดงผล / Typo
• แก้ไขไฟล์เดี่ยวที่พี่พีชระบุชัดเจน
• เพิ่ม entry ใน CHANGELOG หลัง Merge | • งานที่กระทบไฟล์มากกว่า 2 ไฟล์ขึ้นไป
• การเปลี่ยน Logic หลัก (ราคา, สต๊อก, ออเดอร์, UID)
• การสร้างไฟล์ใหม่ หรือปรับโครงสร้างโฟลเดอร์
• คำสั่งที่มีความคลุมเครือ หรือไม่แน่ใจ 100% |
### 🛠️ ขั้นตอนการทบทวนงาน (บังคับใช้เมื่องานไม่ชัดเจนหรือมีความเสี่ยง)
หากงานเข้าข่ายต้องรออนุมัติ ให้ส่งข้อความทบทวนกลับตามรูปแบบนี้ก่อนเริ่มเขียนโค้ด:
 1. **สรุปความเข้าใจ:** สรุปสั้น ๆ ว่าเข้าใจว่าอะไร (ใช้ภาษาไทยบ้าน ๆ ห้ามใช้ศัพท์เทค)
 2. **แยกแยะชัดเจน:** ทำเป็น Bullet ว่าอะไรคือสิ่งที่ **[ทำ]** / **[ไม่ทำ]** / **[ถามพี่พีชก่อน]**
 3. **ยกตัวอย่างพฤติกรรม:** ยกตัวอย่างการทำงานหรือการแสดงผลในแอป 1-2 บรรทัดให้เห็นภาพชัดเจน
 4. *รอคำสั่ง "ทำเลย" หรือ "ใช่แบบนี้" จากพี่พีชก่อนเปิด PR เสมอ*

## 6. กระบวนการสืบสวนและแก้ไขบั๊ก (Debugging Flow)
เมื่อพี่พีชแจ้งว่าระบบมีปัญหาหลังจากเพิ่งแก้ไขไป **ห้ามเดาวิธีแก้เองใหม่ทั้งหมด** ให้ทำตามขั้นตอนดังนี้:
 1. **เปิดอ่านประวัติ:** ตรวจสอบไฟล์ docs/AGENT_CHANGELOG_TH.md เพื่อดู Entry ล่าสุดที่เพิ่งแก้ไขไป
 2. **วิเคราะห์จุดเสี่ยง:** อ่านหัวข้อ [แก้แล้ว], [ไฟล์สำคัญ] และ [ถ้าพังอีกให้เช็ก] จากบันทึกรอบก่อน
 3. **ลงมือแก้ไข:** ทำการ Debug ต่อเนื่องจากจุดเดิมเพื่อไม่ให้กระทบส่วนอื่น
 4. **บันทึกประวัติ:** หลังจากทำการ Merge เรียบร้อยแล้ว เอเจนต์ต้องเพิ่ม Entry สรุปสิ่งที่แก้เป็นภาษาไทยสั้น ๆ ในไฟล์ docs/AGENT_CHANGELOG_TH.md ทุกครั้ง

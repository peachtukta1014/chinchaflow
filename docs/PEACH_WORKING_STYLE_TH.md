# แนวทางสื่อสารกับ Peach (สำหรับ AI Agent)

ให้เอเจนต์อ่านก่อนลงมือทุกครั้ง — นี่คือบริบทที่สำคัญที่สุดในการทำงานร่วมกับ Peach

## ตัวตนพีช + บริบทโปรเจกต์ (อ่านก่อนทุกอย่าง)

- Peach คือ **เจ้าของธุรกิจ 2 ร้าน**: ชินชา (ร้านชา) + โกอ้วน (ร้านกุ้ง/ซีฟู้ด)
- **ความรู้โปรแกรมเมอร์ = ศูนย์** ตั้งแต่เริ่มต้น — AI คือคนเดียวที่เขียนโค้ดในโปรเจกต์นี้ทั้งหมด
- **ไม่มีเครื่องคอมในกระบวนการทำงานเลย** — ทุกอย่างรันบน cloud และมือถือ 100%
- **ไม่มีทีม** — ทำคนเดียว ระหว่างดูแลร้านจริงในชีวิตประจำวัน

**ประวัติโปรเจกต์ (สำคัญ — AI ต้องเข้าใจ):**
- 3 เดือนแรก: พยายามรันโค้ดบนเครื่องคอม (laptop) → โค้ดพังบ่อย ล้มเหลวมาก
- ลูกชายคนเล็กแนะนำ Cursor → ใช้เวลา 2 สัปดาห์จนเจอทางทำงานผ่านมือถือได้
- 2 สัปดาห์สุดท้าย (ปัจจุบัน): โปรเจกต์เดินหน้าจริง
- **รวม 4 เดือน** กับโค้ดกว่า **60,000 บรรทัด** ที่สร้างจากศูนย์ด้วยความพยายามสูงมาก

→ **เอเจนต์ต้องระวังเป็นพิเศษ**: โค้ดทุกบรรทัดมีค่า อย่า break สิ่งที่ทำงานอยู่ diff เล็กสุดเสมอ

## ทำงานผ่านมือถือ 100% (ไม่มีข้อยกเว้น)

- **ไม่มีเครื่องคอมในกระบวนการเลย** — ไม่มี terminal, ไม่มี IDE, ไม่มี diff viewer บน PC
- Peach คุยกับ AI ผ่าน **ai-chat PWA** (chincha-ai-chat.web.app) และ Claude Code บนมือถือ
- ระบบที่ใช้ปัจจุบัน: **Claude Code + OpenRouter (DeepSeek v4 Flash/Pro) + GitHub API**
- เอเจนต์ต้อง: **สรุปสั้น ชัด อ่านบนมือถือได้** · ลิงก์ PR เป็นหลัก · ไม่ให้ดู diff ยาวหรือรัน command
- Peach ยืนยันงานในแชตด้วยคำสั้นๆ: «โอเค», «ทำเลย», «เปิด PR ได้เลย»

## บริบทผู้สั่งงาน

- Peach สั่งงาน **ภาษาพูดทั่วไป** — ไม่รู้ศัพท์ dev (PR, Firestore, webhook, branch, component ฯลฯ)
- มักพิมพ์หรือ **เสียงไมค์** → คำอาจเพี้ยน (ตัวเลข, คำเทค) — อย่าตีความตัวเลขแปลกๆ โดยไม่ถาม
- AI ต้องแปลภาษาชาวบ้านของ Peach → ภาษาเทคนิค เอง ไม่ให้ Peach ต้องรู้ชื่อไฟล์หรือ scope

## ทำได้เลย vs ต้องทบทวน+รอยืนยัน

| ทำได้เลย | ต้องทบทวน + รอ Peach ยืนยัน |
|---------|--------------------------|
| แก้ text / ข้อความ / typo | งานที่กระทบ >2 ไฟล์ |
| แก้ 1 ไฟล์ที่ Peach ระบุชัด | เปลี่ยน logic หลัก (ราคา, สต๊อก, ออเดอร์, UID) |
| เพิ่ม entry CHANGELOG หลัง merge | สร้างไฟล์ใหม่ / โครงสร้างใหม่ |
| | งานที่เข้าใจคำสั่งไม่แน่ใจ 100% |

## หน้าที่ AI ก่อนเขียนโค้ด (บังคับเมื่องานไม่ชัด)

เมื่อคำสั่งเป็นไอเดีย / นโยบาย / หลายประเด็น หรือไม่แน่ใจว่าเข้าใจถูก:

1. **ทบทวนกลับเป็นข้อความสั้น ๆ** ว่าเข้าใจว่าอะไร (ภาษาไทยชาวบ้าน ไม่ใช้ศัพท์เทค)
2. แยกเป็น bullet: **ทำ** / **ไม่ทำ** / **ถาม Peach ก่อน**
3. ถ้ามีคำทับ ให้ยกตัวอย่างพฤติกรรมในแอป 1–2 บรรทัด
4. **รอ Peach ยืนยัน** (“ใช่แบบนี้”, “ทำเลย”) ก่อนเปิด PR — ยกเว้น Peach บอกชัดว่า “ทำเลย”

ตัวอย่างทบทวน:

> เข้าใจว่า: ร้านหนึ่งมี LINE ได้หลายคน — **คนโอน/เจ้าของ** รับบิลอย่างเดียว · **คนสั่ง** ใน LINE ให้ระบบผูกร้านอัตโนมัติเมื่อสั่งในร้านเดิม · **ไม่** ปิดรับออเดอร์ตามเวลา · ร้าน 2 สาขา = **2 แถวในรายชื่อลูกค้า** แยกบิลตามชื่อร้าน — ถูกไหมครับ?

## เมื่อสั่งแก้จุดหนึ่ง แล้วมีปัญหาทีหลัง

Peach ต้องการให้เอเจนต์ **รู้ว่ารอบก่อนแก้อะไรไปแล้ว** — ไล่บั๊กจากจุดนั้นก่อน

| ขั้น | ทำอะไร |
|------|--------|
| 1 | เปิด `docs/AGENT_CHANGELOG_TH.md` หา entry ที่เกี่ยว |
| 2 | อ่าน「แก้แล้ว」「ไฟล์สำคัญ」「ถ้าพังอีกให้เช็ก」 |
| 3 | ค่อย debug / แก้ต่อ — อย่าเดาใหม่ทั้งก้อนถ้ายังไม่ได้อ่านรอบก่อน |
| หลัง merge | เอเจนต์ **เพิ่ม entry** สรุปว่าแก้อะไร (ภาษาไทย สั้น ๆ) |

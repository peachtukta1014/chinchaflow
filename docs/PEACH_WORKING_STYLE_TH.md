# แนวทางสื่อสารกับ Peach (สำหรับเอเจนต์ / พี่เซอ)

อัปเดตเมื่อทีมตกลงแนวทาง — ให้เอเจนต์อ่านก่อนลงมืองานจาก Slack หรือ Cursor Cloud

## บริบทผู้สั่งงาน

- Peach สั่งงาน **ภาษาพูดทั่วไป** มากกว่าศัพท์ dev (ไม่จำเป็นต้องรู้คำว่า PR, Firestore, webhook, branch)
- มักพิมพ์หรือ **เสียงไมค์** → คำอาจเพี้ยน (ตัวเลข, คำเทค) — อย่าตีความตัวเลขแปลกๆ เป็นชื่อร้านโดยไม่ถาม
- ช่อง Slack กุ้ง: `#chincha-shrimp-agent` · ชา: `#chincha-tea-agent`

## หน้าที่พี่เซอก่อนเขียนโค้ด (บังคับเมื่องานไม่ชัด)

เมื่อคำสั่งเป็นไอเดีย / นโยบาย / หลายประเด็น:

1. **ทบทวนกลับเป็นข้อความสั้น ๆ** ว่าเข้าใจว่าอะไร (ภาษาไทย ไม่ยืด)
2. แยกเป็น bullet: **ทำ** / **ไม่ทำ** / **ถาม Peach ก่อน**
3. ถ้ามีคำทับ (เช่น “บันทึก UID” vs “ส่งบิล”) ให้ยกตัวอย่างพฤติกรรมในแอป 1–2 บรรทัด
4. **รอ Peach ยืนยัน** (“ใช่แบบนี้”, “ทำเลย”) ก่อนเปิด PR ใหญ่ — ยกเว้น Peach บอกชัดว่า “ทำเลย” / “เปิด PR ได้เลย”

ตัวอย่างทบทวน:

> เข้าใจว่า: ร้านหนึ่งมี LINE ได้หลายคน — **คนโอน/เจ้าของ** รับบิลอย่างเดียว · **คนสั่ง** ใน LINE ให้ระบบผูกร้านอัตโนมัติเมื่อสั่งในร้านเดิม · **ไม่** ปิดรับออเดอร์ตามเวลา · ร้าน 2 สาขา = **2 แถวในรายชื่อลูกค้า** แยกบิลตามชื่อร้าน — ถูกไหมครับ?

## คำศัพท์ที่ Peach มักใช้ ≈ ความหมายในระบบ

| Peach พูดประมาณ | ใน repo / แอป |

## เมื่อสั่งแก้จุดหนึ่ง แล้วมีปัญหาทีหลัง

Peach ต้องการให้เอเจนต์ **รู้ว่ารอบก่อนแก้อะไรไปแล้ว** — ไล่บั๊กจากจุดนั้นก่อน

| ขั้น | ทำอะไร |
|------|--------|
| 1 | เปิด `docs/AGENT_CHANGELOG_TH.md` หา entry ที่เกี่ยว |
| 2 | อ่าน「แก้แล้ว」「ไฟล์สำคัญ」「ถ้าพังอีกให้เช็ก」 |
| 3 | ค่อย debug / แก้ต่อ — อย่าเดาใหม่ทั้งก้อนถ้ายังไม่ได้อ่านรอบก่อน |
| หลัง merge | เอเจนต์ **เพิ่ม entry** สรุปว่าแก้อะไร (ภาษาไทย สั้น ๆ) |
